### 1.3 CLI contract (from `cmd/docs_chain/main.go`, read directly from the repo)


Exit codes: `0` in-sync/applied/healthy · `1` generic error · `2` sync

gate)"* — this is the exact primitive §11.4.50 (deterministic PASS/FAIL/SKIP)
and the pre-build seam this design wires in.

### 1.4 Context schema (`.docs_chain/contexts/*.yaml`, from `docs/CONFIG_SCHEMA.md` + the shipped `self-docs.yaml` example)

Top-level: `context` (name) / `description` / `nodes` (map of `node_id →
{kind, path[, members, exclude]}`) / `edges` (list of `derive-from` or `sync`
edge-specs) / `transforms` (map of `name → {builtin: ...}` or `{exec: ...,
args: [...]}`).

Node kinds relevant here: `markdown` (canonical source, input) / `html`
(pandoc export, derived) / `pdf` (weasyprint export, derived) / `fingerprint`
(roster/corpus member-list hash, §11.4.86 — not needed for this design, no
roster/corpus doc in the tracked set). `docx` is available via the
`pandoc-docx` builtin but is **not** part of the §11.4.65 mandate (only the
§11.4.153-class four-format mandate needs it) — left as an optional future
extension, not required here.

Builtins used: `pandoc-html` (`markdown → html`, needs `pandoc`),
`weasyprint-pdf` (`html → pdf` or `markdown → pdf`, needs `weasyprint`).

The engine's own dogfood context (`vasic-digital/docs_chain`'s own
`.docs_chain/contexts/self-docs.yaml`) is the exact worked pattern this
project's context should follow — one `markdown` node, one `html` node
derived from it via `pandoc-html`, one `pdf` node derived from the `html`
node via `weasyprint-pdf`, repeated per tracked document:

```yaml
- { type: derive-from, from: readme_md,  to: readme_html, transform: md2html }
- { type: derive-from, from: readme_html, to: readme_pdf,  transform: html2pdf }
transforms:
  md2html:  { builtin: pandoc-html }
  html2pdf: { builtin: weasyprint-pdf }
```

### 1.5 Content-hash detection, not mtime (§11.4.86 compliance confirmed)

`docs/CONSTITUTION_INTEGRATION.md` (fetched from the repo) lists "Content-hash
change detection (§11.4.86 not-mtime)" as a for-free, engine-wide property —
`state.json` records content hashes (`internal/state`, `internal/hash`), and
`verify` recomputes + compares bytes rather than trusting a timestamp. This is
exactly the property §11.4.65/§11.4.86 require and precisely what an ad-hoc
`find -newer`-style sync script would get wrong.

### 1.6 Honest tool-absence path (§11.4.106(E)) — confirmed present in the source, not assumed

`internal/adapter/adapter.go` defines a typed `ToolAbsentError` whose doc
comment states verbatim: *"It is NEVER a fake-success: callers (and tests)
match on it via `errors.As` to SKIP-with-reason rather than claim a transform
ran."* Its `Error()` string is explicit: `"adapter: required tool %q not
found ...; refusing to fake success"`.

`docs_chain doctor` proactively resolves `pandoc`/`weasyprint` on `PATH` per
transform and emits a `WARN` (never a hard failure) when absent — *"transform
%q needs %q (not on PATH) — runs will SKIP-with-reason"*.

`docs_chain verify`'s per-context output is a three-way, not a two-way,
result: `in-sync` / `STALE: [...]` (a real drift, gate-blocking) / `SKIP (tool
absent): <reason>` (honest, non-blocking) — the `ToolAbsent` bool is a
first-class field on the verify result, so a pre-build gate wrapper never has
to string-match to tell a real staleness apart from an honest tool-absence
skip.

`docs_chain sync`'s rollback path is atomic (temp-file write → `fsync` →
`rename(2)` commit phase; any error before commit deletes all temps and
aborts — `docs/ARCHITECTURE.md` §8, "Atomicity & crash-safety, composes with
§9.2") — a tool-absent transform never leaves a partial/fake export on disk.

**One nuance for the gate wrapper design (§3 below):** `sync`'s own
`exitForStatus` maps every `StatusRolledBack` (both a genuine transform
failure and an honest tool-absent skip) to the same exit code `3`; the two
are distinguished only in the human-readable message
(`"ROLLED-BACK: SKIP (tool absent): ..."` vs `"ROLLED-BACK: transform failed: ..."`).
`verify`, by contrast, already separates `SKIP (tool absent)` from `STALE` as
distinct, directly-testable result states. **This design uses `verify` — never
`sync` — as the pre-build gate** specifically because `verify` does not
conflate the two; `sync` is reserved for the explicit, human/CI-invoked
regeneration step (§3.2), where a mixed exit code is acceptable because a
human or a dedicated step is already looking at the log line.

## 3. Design

### 3.1 Consumer-owned context config (`.docs_chain/contexts/*.yaml`)

Per §11.4.28/§11.4.177, the engine is inherited by reference and carries zero
project literals; every path lives in project-owned YAML under
`<repo_root>/.docs_chain/contexts/`. Proposed split into **two** contexts
(not one monolith), matching the natural document-lifecycle boundary already
visible in the tree:

1. **`skilltree_core_docs.yaml`** — the six named "living" documents that
   change frequently and are read first by a resuming session
   (`REQUIREMENTS.md`, `GAPS_AND_RISKS_REGISTER.md`, `CONTINUATION.md`,
   `SPEC.md`, `IMPLEMENTATION_PLAN.md`, `requests/history.md`, plus
   `plan.md`). Each gets one `markdown` node, one `html` node
   (`pandoc-html`), one `pdf` node (`weasyprint-pdf`), following the
   `self-docs.yaml` pattern exactly (§1.4).
2. **`skilltree_research_notes.yaml`** — the ~40 `research/*.md` /
   `project/docs/*.md` design notes (append-mostly, read far less often).
   Same per-doc `derive-from` triple, just a longer node/edge list.

Rationale for the split: `docs_chain sync <context>` / `verify <context>`
operate per-context, so splitting lets a fast pre-build gate check only the
core-docs context on every build while the (larger, slower-changing) research
notes context is checked on a coarser cadence (e.g. release-gate sweep,
§11.4.40) without inventing a new mechanism — it is just `--all` vs a named
context, which the CLI already supports. This is a recommendation, not a hard
requirement; a single combined context is equally correct (no cycle risk
either way — every edge is one-directional `derive-from`, `md → html → pdf`,
never a `sync`/bidirectional edge for this doc class).

Both files are **data**, own-project, never engine code (§11.4.28-B).
`.docs_chain/state.json` and `*.docs_chain.tmp` are added to `.gitignore`
(mirrors the constitution's own precedent — `state.json` is regenerated by
`sync`, not authored).

### 3.2 Wiring seams

- **`sync` seam (regeneration):** a project-owned helper
  (`scripts/sync_docs_chain.sh`, thin wrapper analogous to
  `constitution/scripts/codegraph_sync.sh`) invokes `docs_chain sync --all`
  from the repo root. This is the seam that actually **writes** the `.html`/
  `.pdf` files. Composes with §11.4.106(F) write-seam enforcement: it runs (a)
  at the commit seam — before a commit touching a tracked source `.md`, so the
  derived exports never drift behind what gets committed (mirrors the
  existing `sync_issues_docs.sh`-class pattern this Constitution already
  mandates for the Issues/Fixed family); (b) on demand via the helper.
- **`verify` seam (gate):** a pre-build gate (`CM-DOCS-CHAIN-VERIFY`, landed
  under `constitution/scripts/gates/` alongside the project's other gates)
  runs `docs_chain verify --all` read-only. Per §11.4.50 it is deterministic:
  `exit 0` when every context reports `in-sync` (or an honest `SKIP (tool
  absent)`), non-zero when any context reports `STALE`. This is the gate a
  commit or a build refuses to proceed past.
- Both seams read `.docs_chain/contexts/*.yaml` — no project literal lives in
  the engine binary itself.

### 3.3 Honest tool-absence path in this design (§11.4.106(E))

On **this** host, `pandoc` (3.10) and `weasyprint` (69.0) are **both present**
and resolve via `command -v` (confirmed directly, not assumed — see §4). So in
normal operation on this host, `verify`/`sync` will exercise the real
transforms, not the SKIP path. The honest-SKIP path is nonetheless part of
the design (§1.6) because: (a) it is what makes the gate portable to a future
host/CI runner that lacks these tools without ever risking a fake pass, and
(b) the paired §1.1 mutation proving the gate is load-bearing (§3.4) should
also prove the SKIP path fires correctly and is never mistaken for a `STALE`
or an `in-sync` result. If pandoc/weasyprint were absent on a given host, the
correct behavior is a typed, honest `SKIP (tool absent): <reason>` per
context — never a partial export, never a silent pass — with a tracked
follow-up to install the tool (§11.4.184-style pattern: install once,
durably, on `PATH`).

### 3.4 Test plan

1. **`docs_chain doctor --all`** (no writes) — validates both context files
   parse, their `derive-from` sub-graphs are acyclic, and reports per-transform
   tool availability. Expected on this host: `transforms: OK (all required
   tools present)` for both contexts (since pandoc + weasyprint resolve).
2. **`docs_chain verify --all`** as the deterministic pre-build gate
   (`CM-DOCS-CHAIN-VERIFY`). Golden-good: after a `sync`, `verify` reports
   `in-sync` for every context, exit 0. Golden-bad / paired §1.1 mutation:
   after a `sync`, hand-edit one tracked source (e.g. append a line to
   `REQUIREMENTS.md`) **without** re-running `sync` — `verify` MUST report
   `STALE: [...]` naming the dirty node, and the gate MUST exit non-zero (1).
   Revert the edit (or re-`sync`) — `verify` MUST return to `in-sync`, exit 0.
   This is the RED→GREEN polarity proof (§11.4.115) that the gate is real,
   not a tautology.
3. **ToolAbsentError honest-SKIP proof:** run `docs_chain doctor --all` (and
   separately `sync`/`verify`) with `PATH` scoped to exclude both tool
   directories (`/home/milos/Factory/software/pandoc/bin` and
   `/home/milos/Factory/software/weasyprint/bin` on this host). `doctor` MUST
   emit the `WARN ... not on PATH — runs will SKIP-with-reason` line for both
   transforms; `sync` MUST roll back with a `ROLLED-BACK: SKIP (tool absent)`
   message (no partial file written — confirmed by the atomic
   temp→fsync→rename commit semantics in §1.6, so a rolled-back sync leaves
   zero new bytes on disk); `verify` MUST report `SKIP (tool absent): ...`
   distinctly from `STALE`, and this SKIP MUST NOT be counted as a gate
   failure (composing §11.4.3's SKIP-with-reason ≠ FAIL discipline).
4. **Real-export proof (once tooling-gated wiring lands):** after the first
   real `sync --all`, each of the 46+ tracked docs has a byte-present,
   non-empty `.html` and `.pdf` sibling; a second `verify --all` immediately
   after reports `in-sync` for all (no false-STALE flapping — proves
   determinism, §11.4.50).

## 5. Revision-header dependency (G44 coupling)

Four of the six named core docs — `REQUIREMENTS.md`, `GAPS_AND_RISKS_REGISTER.md`,
`SPEC.md`, `IMPLEMENTATION_PLAN.md` — have **no** §11.4.44 revision header in
their first 8 non-blank lines (confirmed directly); `CONTINUATION.md` (Rev 7)
and `requests/history.md` (Rev 1) already do. Docs Chain does **not** enforce
or author the §11.4.44 header itself — per `docs/CONSTITUTION_INTEGRATION.md`
§4 (fetched from the repo): *"Docs Chain does NOT replace the authoring
discipline of any anchor (e.g. the operator still writes the §11.4.44
revision header into the source); it replaces the **mechanical sync** the
anchor requires."* The header is the source author's responsibility; the
engine only keeps the derived `.html`/`.pdf` **in sync with whatever the
source currently says**, header or no header.

**Coupling, not a hard blocker:** G43 (this wiring) and G44 (the header
backfill) can land in either order without breaking each other — a
Docs-Chain-exported `.html` of a header-less `REQUIREMENTS.md` is still a
faithful, in-sync export; it is simply missing the header the source itself
is missing. That said, landing G44 **first or in the same batch** is
recommended so that the first real `sync --all` (§3.4 step 4) produces
exports that already carry the header, avoiding a second wasted re-sync
purely to pick up the header text. This is a sequencing preference (§11.4.42
iteration discipline), not a technical dependency.

## 6. Sequencing (§11.4.101 — a separate, low-disruption, gated step)

This design recommends landing the wiring **after** the P0.5 critical-path Go
spine remediation the project's `REQUIREMENTS.md` describes as the current
phase, as its own gated step, for two reasons: (a) it touches zero source
code relevant to the Go spine — it is purely documentation-export tooling,
so there is no reason to contend with in-flight P0.5 work for reviewer
attention; (b) vendoring a new submodule (`constitution/submodules/docs_chain/`,
following the existing `continuum`/`session_orchestrator`/`token_optimizer`
pattern) is itself a §11.4.36/§9.2-class action an operator should confirm,
not something silently bundled into an unrelated fix.

Two distinguishable states, so "wired" is never conflated with "exports
exist" (§11.4.6):

- **State A — "engine wired + verify-gate green (exports SKIP honestly if
  tooling absent)."** The submodule (or a by-reference sibling checkout, per
  the upstream doc's own interim pattern — Docs Chain's Phase 6
  constitution-submodule distribution is itself still upstream-PLANNED and
  operator-gated, so this project's adoption today mirrors how the first
  consumer, Herald, adopted it: build the binary from a referenced checkout,
  not wait for a turnkey submodule pointer) is present and buildable, the two
  context YAMLs exist, `CM-DOCS-CHAIN-VERIFY` is wired into the pre-build
  path, and `verify --all` runs cleanly — reporting `STALE` for every node
  (because no `sync` has run yet) or an honest `SKIP` if tooling were absent.
  This state proves the mechanism without yet claiming any document has a
  fresh export.
- **State B — "exports actually generated."** The first `sync --all` has run,
  every tracked doc has a real `.html`+`.pdf` sibling, and `verify --all`
  reports `in-sync` across the board. This is the state that actually closes
  G43's "zero exports exist" finding.

Recommended order: land State A fully gated and reviewed first (cheap,
reversible, proves the plumbing with zero export side-effects); then run the
first `sync --all` and commit the generated siblings as State B, in the same
change-window or the next one, per operator confirmation.

  confirm empirically once the engine is actually invoked in this repo).
- DOCX siblings are optional and out of scope for the §11.4.65 baseline
  mandate; not designed further here.
- No `.gitmodules`/gate-script/context-YAML file was created or modified —
  this document is the only artifact this task produced.

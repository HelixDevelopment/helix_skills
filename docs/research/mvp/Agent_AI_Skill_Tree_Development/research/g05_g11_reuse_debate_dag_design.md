1. **G05 (LLM jury) on `debate_orchestrator` — STUDY-only.** The fail-closed two-factor consensus the jury needs (`approvals >= threshold && votes >= juryMinApprovals`) is **already implemented and correct** in the project's own `LLMJury`/`juryStage`/`computeOverallVerdict` (pipeline.go:420-439, 445-516; sandbox.go:59-79). `debate_orchestrator`'s headline `Orchestrator.ConductDebate` computes a **different, weaker** consensus (a single blended `avgConfidence >= MinConsensus` float threshold, orchestrator.go:338) — adopting it would be a regression, not a reuse win. This **refines** (not reverses) `research/helix_family_reusable_practices.md` §6B's "ADOPT-core" verdict: that pass evaluated the module's general shape for "N models vote on a topic" at the README level; this pass verified the exact consensus arithmetic against the real, already-landed G05 fix and found no net win. Do not incorporate for G05. Revisit only if `debate_orchestrator`'s `voting.Tally` graduates to REAL upstream, or if the project later needs genuine multi-round adversarial debate with dissent + lesson persistence — a materially bigger feature than a single-round jury vote.

2. **G11 (worker scheduling) on `dag_orchestrator` — INCORPORATE-as-submodule (partial, scoped).** `dag_orchestrator`'s `Scheduler.Run` (dag.go:185-344) is a **complete, tested, race-clean** ready-set dispatcher with a parallelism cap, a `FailFast`/`ContinueOnError` failure policy, per-node `RetryPolicy`, **per-node panic recovery** (`runOnce`, dag.go:397-405), and dynamic fan-out via `Expandable` (funcnode.go:25-31) — a direct, non-regressive replacement for the *scheduling* half of `g11_worker_design.md`'s two coupled defects (runner.go:509-527 unchecked assertions; runner.go:221/375/396/414 recover-less goroutines). It does **not** replace the ticker-driven perpetual polling loops, the Postgres single-owner advisory lock, or any skill-graph business logic — those stay project-specific and largely as `g11_worker_design.md` already designed them. This **confirms and narrows** `research/helix_family_reusable_practices.md` §8A's one-line flag ("direct candidate for G11 worker scheduling... a follow-on pass should design its wiring") — this document IS that follow-on pass.

## 1. G05 — LLM validation jury on `digital.vasic.debate` (debate_orchestrator)

### 1.1 FACT summary: what the vendored repo actually provides

Verified from source, not the README alone:

- **Module identity.** `module digital.vasic.debate`, Go 1.26, **stdlib-only** (`debate_orchestrator/go.mod:1-3`; `debate_orchestrator/helix-deps.yaml:1-8` — `deps: []`, "inspected go.mod; own-org deps: none").
- **Root package `debate`** (`debate_orchestrator/debate.go:1-216`): a thread-safe in-memory `LessonBank` (CRUD + substring `Search`) plus session-metadata accessors (`SetSession`, `Status`, `Conclusion`). REAL, minimal, no persistence.
- **`orchestrator` package** (`debate_orchestrator/orchestrator/orchestrator.go:1-534`, `types.go:1-371`, `pool.go:1-58`): `Orchestrator.ConductDebate(ctx, *DebateRequest) (*DebateResponse, error)` runs `resolved.MaxRounds` rounds over an `AgentPool` (`RegisterProvider`/`selectParticipants`), building one `AgentResponse` per agent per round via `invokeAgent` (orchestrator.go:455-471). When a `ProviderInvoker` is wired via `WithProviderInvoker` (orchestrator.go:44-58), `invokeAgent` calls it with `(ctx, prompt string)` and measures real wall-clock latency (`time.Since(start)`, orchestrator.go:460-462); when unwired, it falls back to a deterministic, **self-labelled** synthesised-content stub (`synthesiseContent`, orchestrator.go:499-506, marker string `"[synthesised ... awaiting provider wiring]"`).
- **Consensus mechanism (the one that matters for this task).** `ConductDebate` computes `avgConfidence` as the mean of every `AgentResponse.Confidence` across every round and agent (orchestrator.go:328-331), and sets `ConsensusResponse.Achieved = avgConfidence >= resolved.MinConsensus` (orchestrator.go:338, default `MinConsensus = 0.6` at orchestrator.go:404-406 if the caller omits it). `Confidence` per response is `scoreToConfidence(agent.Score, round)` (orchestrator.go:508-519) — a deterministic function of the agent's static `Score` field and the round index, **not** a parsed approve/reject decision from the LLM's actual output. There is **no** `Approved bool` field anywhere on `AgentResponse` (types.go:127-148) and no per-juror approval count anywhere in `ConductDebate`'s output.
- **`voting` package** (`debate_orchestrator/voting/voting.go:1-616`): a *separate* vote-tally surface, not wired into `ConductDebate` at all (no `orchestrator` package import of `voting`, confirmed by reading `orchestrator.go`'s import block, orchestrator.go:9-15). Real, source-verified facts:
  - `Vote{AgentID, Choice, Confidence, Score, Specialization, Role, Timestamp}` (voting.go:93-109); `WeightedVotingSystem.AddVote`/`VoteCount`/`GetStatistics`/`Reset` are REAL (voting.go:184-224).
  - `Calculate` (voting.go:249-294) — REAL: weighted-confidence-sum winner-take-all over `AddVote`'d votes, gated by `Config.MinimumVotes` (voting.go:256) but with **no per-choice count returned as a first-class field** (the winner's raw vote count must be derived by the caller from `Tally[winner]` only when `Calculate`'s summation happens to equal a count, which it does not — `Calculate` sums `Confidence`, not `1` per vote).
  - `CalculatePlurality` (voting.go:491-558) — REAL: this one **does** tally raw counts (`tally[v.Choice]++`, voting.go:507), with `EnableTieBreaking`/`TieBreakMethod` support (voting.go:534-547). This is the closest REAL primitive to "count of approvals."
  - `CalculateUnanimous`/`CalculateBordaCount`/`CalculateCondorcet` — REAL per the package doc comment (voting.go:1-21) and confirmed present (voting.go:307, 378, 568).
  - `Tally(ctx, votes []Vote) (string, error)` (voting.go:226-238) — the literal one-shot "give me a winner" entry point named in the package doc as the module's primary API surface — is an **honest, explicit stub**: `return "", errors.New("debate/voting: Tally NotYetImplemented — see RECONSTRUCTION_ROADMAP.md")` (voting.go:237), `TODO(reconstruction-phase-2)` (voting.go:232).
  - `Config.MinAgents int` (voting.go:63-64, "the minimum number of agents required to tally") is **declared and defaulted** (`DefaultVotingConfig`, voting.go:172, `MinAgents: 2`) but **never read** anywhere in `voting.go` — confirmed by grepping every `MinAgents` occurrence in the package: only the field declaration and the default-value assignment exist; no `Calculate*` method, `AddVote`, or any other function checks `s.Config.MinAgents` against `len(s.votes)` or `VoteCount()`. This is dead configuration, not a documentation gap — a caller wanting "votes >= min" enforcement must implement it themselves by comparing `sys.VoteCount()` (or a `Result.TotalVotes`) to their own floor.
- **`gates` package** (`debate_orchestrator/gates/gates.go:1-47`): `ApprovalGate.Check` is a REAL but permissive baseline — it **always** returns `(true, nil)` whether `Enabled` is true or false (gates.go:38-47, "wired but never blocking"). Real policy evaluation is an explicit follow-up (RECONSTRUCTION_ROADMAP.md row `gates`).
- **Package-tier honesty (self-disclosed, `debate_orchestrator/README.md:45-61`):** REAL — `debate`(root), `agents`, `topology`, `gates`, `orchestrator`, `comprehensive`(core; streaming stubbed). STUB (constructors real, every execution method an explicit `NotYetImplemented` error, never silent success) — `validation`, `audit`, `evaluation`, `reflexion`, `tools`, `testing`.
- **A documentation-vs-source discrepancy worth flagging as its own fact (§11.4.6 — state it, don't guess which is "right"):** the root `README.md`'s "Package tier table" (README.md:45-61) **omits the `voting` and `protocol` packages entirely** — they are not rows in that table. `RECONSTRUCTION_ROADMAP.md`'s separate per-package table (RECONSTRUCTION_ROADMAP.md:10-25) **does** list them, both marked `STUB (test-only, mostly stub)`. Reading `voting.go`'s own package doc comment and its actual function bodies (above) shows this ROADMAP characterization is **stale relative to the current source**: five of six public tally methods (`Calculate`, `CalculateBordaCount`, `CalculateCondorcet`, `CalculatePlurality`, `CalculateUnanimous`) are REAL, and only `Tally` itself is the honest stub. Both facts are recorded here rather than reconciled by assumption; the source (verified by direct read + grep) is the higher-confidence signal, so this design treats `voting` as "mostly REAL, one stub method" rather than "STUB."

### 1.2 Fit analysis against the skill-graph's ACTUAL current G05 code

The task's premise — evaluate whether `debate_orchestrator` fits "N jurors vote/debate → consensus, the two-factor `approvals>=threshold && votes>=min`" — must be checked against what the skill-graph project **already has**, not what `g11_worker_design.md` (written before the G05 fix landed) assumed was still pending. Verified facts about the current tree (HEAD, not `255061b`):

- `internal/validation/pipeline.go:23-26` — `const juryMinApprovals = 2` — "the fail-closed floor on real approvals required for the jury stage to reach consensus (§G05)."
- `pipeline.go:418-439` (`juryStage`) and `pipeline.go:441-516` (`LLMJury`) already implement **exactly** the two-factor rule the task describes: `consensus := approvalCount >= juryThreshold(p.cfg) && len(votes) >= juryMinApprovals` (pipeline.go:502). An empty jury (`len(p.jury) == 0`) is a hard `StageBlocked` fail-closed verdict, never an auto-pass (pipeline.go:431-433, 446-455).
- `sandbox.go:59-79` (`computeOverallVerdict`) implements the pipeline-wide fail-closed aggregation the task cites: PASS only if every stage is `StagePass` or `StageNA`, `StageSkip`/`StageFail`/`StageBlocked` never upgrade to an overall PASS.
- The jury's LLM-calling mechanism is **already abstracted** behind `LLMValidator` (`ValidateSkill(ctx, *models.Skill) (approved bool, feedback string, err error)`, pipeline.go:59-63) and `JuryMember{Name, Weight, LLM}` (pipeline.go:52-57), dispatched **concurrently** via a plain `sync.WaitGroup` fan-out with per-juror error handling (pipeline.go:464-491) — this is the "provider layer" the task instructs to treat as an injected dependency, and it is already correctly wired to accept N independent `LLMValidator` implementations (i.e., N distinct real LLM backends), one per `JuryMember`.
- **The actual remaining gap is NOT the consensus math — it is that no real `JuryMember` is ever constructed.** `WithJury` (pipeline.go:98-103) is defined but is **never called anywhere in production code**: grepping the whole `project/` tree for `WithJury(` and `JuryMember{` finds only the definition itself in `pipeline.go`; `cmd/worker/main.go` and `cmd/server/main.go` never call it. Confirmed further: `validation.NewPipeline` and `autoexpand.NewPipeline` are **never called at all** in `cmd/worker/main.go` (only `worker.NewRunner(pool, store, *cfg, logger)` at `cmd/worker/main.go:94`) — this is the exact G11 gap #1 (`g11_worker_design.md` §0/§2.3) independently reconfirmed here. `pipeline_test.go:121` (`TestLLMJury_EmptyJuryBlocks`) and `pipeline_test.go:155` (`TestLLMJury_RequiresTwoApprovals`) prove the fail-closed math is unit-tested and correct; nothing tests a real end-to-end N-provider jury today because nothing constructs one.

So the real decision this task poses reduces to: **is `debate_orchestrator` a good source for the missing piece — "N real LLM juror calls, dispatched, with captured latency and explicit error propagation" — that would sit *underneath* the already-correct `LLMJury`?** Checked directly:

- `Orchestrator.ConductDebate`'s `ProviderInvoker` callback signature is `func(ctx context.Context, prompt string) (string, error)` (types.go:262-264) — it receives **only the prompt text**, not the calling `*Agent` (no `agent.Provider`/`agent.Model`/`agent.ID` parameter). `invokeAgent` (orchestrator.go:455-471) calls `o.invoker(ctx, prompt)` — the **same single wired invoker** — for **every** agent in the pool regardless of that agent's `Provider`/`Model` field. There is no per-agent invoker registration anywhere in the package (confirmed: `Option`/`WithProviderInvoker` sets one `o.invoker` field, orchestrator.go:26-42; `AgentPool`/`pool.go:1-58` has no invoker association). **This is the load-bearing gap for G05's actual need**: a genuine multi-model jury requires N *different* real LLM backends to be called, keyed by which juror is being invoked — `ConductDebate`'s API cannot express that without the injected closure parsing `agent.Provider`/`agent.Model` back out of the prompt string it receives (`buildAgentPrompt` embeds `provider=%s model=%s` as prose, orchestrator.go:483-484), which is a fragile string-parsing coupling, not a typed API contract. The skill-graph's own `[]JuryMember{Name, Weight, LLM LLMValidator}` (pipeline.go:52-57) already gives each juror its own concrete `LLMValidator` implementation — a strictly better fit for "N distinct real backends" than `ConductDebate`'s single-invoker model.
- Using `ConductDebate` at all would also pull in machinery the jury does not need and that is either irrelevant or unhelpfully different: multi-**round** debate (`MaxRounds`, `PhaseResponse` per round) when the jury needs exactly one round of independent votes; `Consensus.Achieved` computed from a blended float average (§1.1) instead of a real per-juror approval count; session lifecycle (`CreateSession`/`Session.Status`) and `LessonBank` persistence the jury does not use; `gates.ApprovalGate` that is a permissive no-op today (§1.1) so it adds no real safety; `topology.GraphMesh` participant routing that has no analogue in a flat N-juror vote. None of this is wrong for a *debate* product — it is simply the wrong shape for a *jury vote*.
- Using the separate `voting` package instead of `ConductDebate` is closer in shape (`Vote{Choice, Confidence}` + `AddVote` + `CalculatePlurality`'s real per-choice count, §1.1) but still requires the caller to (a) bypass the honestly-stubbed `Tally` entry point and call the secondary `CalculatePlurality` method directly, (b) implement the "votes >= min" floor themselves since `Config.MinAgents` is dead code, and (c) still construct N real per-juror `LLMValidator`-equivalent calls outside the `voting` package entirely (it has no invocation/dispatch surface of its own — it is a pure tally, not a caller of LLMs). Net: adopting `voting` buys the caller a `CalculatePlurality`-style count-and-tie-break routine in exchange for a new external dependency, in place of the ~10-line raw-count loop the project's own `LLMJury` already has (pipeline.go:425-430, 493-498) — no meaningful simplification, and the "votes>=min" gate still has to be hand-written on top either way.

### 1.3 Verdict: STUDY-only (no incorporation for G05)

**STUDY-only.** Do not incorporate `debate_orchestrator`'s `orchestrator` package, `voting` package, or any other package as a replacement for, or a foundation under, the skill-graph's existing `LLMJury`/`juryStage`/`computeOverallVerdict`. The fail-closed two-factor consensus is already correctly implemented, unit-tested (pipeline_test.go:121, 155), and structurally sound for N independent real `LLMValidator` backends. `debate_orchestrator` offers no consensus primitive that is both (a) REAL/non-stub and (b) a better fit than the code that already exists, and its one API surface that could plausibly host "N real LLM calls" (`ConductDebate`/`ProviderInvoker`) cannot address individual jurors by provider/model without a fragile prompt-string-parsing workaround (§1.2).

This **refines** `research/helix_family_reusable_practices.md` §6B's "ADOPT-as-submodule (`orchestrator` + `debate` core)" conclusion. That pass's own caveat ("the packages the jury most needs for a pass/fail verdict — `validation`/`evaluation` — are STUB") was correct as far as it went, but it evaluated the module against the skill-graph's *future planned* jury (citing `SPEC.md §8 jury_size=3` and `research/helix_interop_incorporation.md §4`) rather than against the jury's *actual, already-landed* implementation, because at the time of that survey the G05 remediation commit (`2befa77`) had not yet landed the two-factor consensus into `pipeline.go`. With the fix now in place, the concrete comparison in §1.2 shows no net benefit to adopting `ConductDebate` or `voting` for this specific mechanism. §6B's broader observation that `debate_orchestrator` is a real, tested, structurally-sound multi-agent coordination module remains true and is not contradicted — it is simply not the right tool for the piece of G05 that remains open.

**What actually remains open for G05** (not this document's job to design, but recorded so it is not lost per §11.4.197): constructing real `JuryMember`s backed by real LLM calls (via whatever `LLMProvider`/`LLMClient` layer R19/G28 lands, per `research/g14_x1_submodule_policy_decision.md:406`) and wiring them into `validation.NewPipeline(..., validation.WithJury(realJury))` at the one call site that needs it (currently absent from both `cmd/server/main.go` and `cmd/worker/main.go`). This is a wiring task against the project's own already-correct interfaces, not a submodule-incorporation task.

## 2. G11 — Background-worker job scheduling on `dev.helix.dag` (dag_orchestrator)

### 2.1 FACT summary: what the vendored repo actually provides

Verified from source:

- **Module identity.** `module dev.helix.dag`, Go 1.26, **stdlib-only** (`context`, `errors`, `fmt`, `sort`, `sync`, `time` — `dag_orchestrator/dag.go:11-18`; confirmed leaf by `dag_orchestrator/helix-deps.yaml:14-15`, `deps: []`, "ZERO own-org submodule dependencies — the scheduler runs on the Go standard library alone").
- **Core interfaces** (`dag_orchestrator/dag.go:27-42`): `Node{ID() string; DependsOn() []string; Execute(ctx, Inputs) (Output, error)}`; `Expandable{Expand(ctx, Output) ([]Node, error)}` (optional, for dynamic fan-out); `Scheduler{Run(ctx, *DAG, Options) (*Result, error)}`; `Build(nodes []Node) (*DAG, error)` validates unique IDs, that every declared dependency exists, and **acyclicity** via an explicit DFS 3-colour cycle check (`findCycle`, dag.go:128-163) — a cycle or a dangling dependency is a real, returned error (dag.go:108-124), never silently accepted.
- **Scheduler.Run** (dag.go:185-344) is a complete, real implementation: builds a `nodeState` per node tracking remaining unmet deps (dag.go:203-214); a bounded semaphore (`sem := make(chan struct{}, par)`, dag.go:216) enforces `Options.Parallelism` (default 1 if `<=0`, dag.go:189-192); `dispatch()` recomputes the ready-set under a mutex (`readyLocked`, dag.go:223-238) and launches one goroutine per ready node; on a node's completion it decrements its dependents' remaining-deps count and re-invokes `dispatch()` (dag.go:284-320) so newly-ready nodes are picked up immediately — this **is** the "ready-set dispatch" the operator's task description names verbatim.
- **Failure policy** (dag.go:43-52, applied at dag.go:285-296): `FailFast` cancels the run context on the first node failure (`aborted = true; cancel()`); `ContinueOnError` instead taints every transitive dependent of the failed node (`taintDependents`, dag.go:372-386) so independent branches still complete — both are REAL, not stubbed, and exercised by `dag_test.go`/`dag_parallelism_test.go` (11 `Test*` functions across the two files, confirmed by `grep -c "^func Test"`).
- **RetryPolicy** (dag.go:54-65, applied via `runWithRetry`, dag.go:407-428): linear backoff, `MaxAttempts` (1 = no retry), context-cancellation-aware between attempts.
- **Per-node panic safety** (`runOnce`, dag.go:397-405): every `Node.Execute` call is wrapped in a `recover()` that converts a panicking node into a normal `error` return (`"dag: node panicked: %v"`) — the scheduler's own doc comment (dag.go:388-396) explains this is deliberately **not** deferred alongside the semaphore release, to avoid deadlocking `dispatch()`'s recursive re-entry at `Parallelism:1`. This is a **direct, working, already-tested instance of exactly the panic-firewall pattern `g11_worker_design.md` §2.2 proposes hand-rolling** (its `supervise()` helper, `g11_worker_design.md:113-137`) — except scoped to a single node's `Execute` rather than an entire long-lived goroutine.
- **Lineage + result shape** (dag.go:74-86): `Result{Outputs map[string]Output, Lineage []Edge, Failed map[string]error, Skipped []string}` — `Edge{From, To}` records "From ran before To" (dag.go:75-78, appended at dag.go:304 on every successful completion). This is real, structured provenance of the traversal order actually taken in a given `Run`, not a static declaration.
- **Dynamic expansion** (`Expandable`, dag.go:39-41, invoked at dag.go:306-315): after a node succeeds, `Expand(ctx, out)` may return additional `Node`s that `addNodes` (dag.go:346-370) wires into the **same live run** — new nodes whose declared deps are already-completed nodes are immediately eligible, and the next `dispatch()` picks them up. `FuncNode` (funcnode.go:1-31) is a convenience `Node`+`Expandable` implementation built from plain functions (`Fn`, `OnExpand`), letting a caller build a DAG without a bespoke type per node — directly usable as the wrapper around each skill-graph work item (see §2.3).
- **Test/challenge maturity.** `dag_test.go` (8 `Test*` functions) + `dag_parallelism_test.go` (3 `Test*` functions), `go test -race -count=1 -v ./...` per README (`dag_orchestrator/README.md:139`); a standalone anti-bluff Challenge (`dag_orchestrator/challenges/dag_orchestrator_challenge.sh`, `cmd/challenge/main.go`) that builds a real diamond DAG `A → {B,C} → D` and asserts from captured runtime evidence that B/C ran concurrently, D ran strictly after both, and `FailFast` gates D + cancels C on a planted failure (`dag_orchestrator/README.md:136-145`). Unlike `debate_orchestrator`, **no package in this module is marked STUB anywhere** in its own docs (`README.md`, `CLAUDE.md`, `CONSTITUTION.md`) — the module's self-description is "Initial scaffold: working in-memory DAG scheduler + 8 passing race tests + anti-bluff Challenge" (`README.md:10`), i.e. narrow in scope but complete for what it claims.

### 2.2 Fit analysis against the skill-graph's ACTUAL current G11 code

`internal/worker/runner.go` and `cmd/worker/main.go` are **byte-identical between `255061b` and HEAD** (confirmed: `git diff 255061b HEAD -- .../internal/worker/runner.go` and `.../cmd/worker/main.go` both produce empty output) — so `g11_worker_design.md`'s analysis of the two coupled defects is still exactly current:

1. **No real work.** `Runner` has only `pool/store/cfg/logger` fields (runner.go:53-64); its cycles (`runAutoExpandCycle`, runner.go:440-481; `runValidationCycle`, runner.go:483-507) only log and increment counters — they never call `autoexpand.Pipeline.Run` or `validation.Pipeline.Validate` (both pipelines exist but are never constructed anywhere in `cmd/worker/main.go`, independently reconfirmed in §1.2 above).
2. **A recover-less panic vector.** `runRegistryReview` (runner.go:509-527) performs two unchecked type assertions (`coverage["total_skills"].(int)`, runner.go:518; `coverage["coverage_percentage"].(string)`, runner.go:519) inside `registryReviewWorker` (runner.go:414-434), a bare `go func` with `defer r.wg.Done()` and **no** `recover()` — matching every other worker loop (`processJobQueue` runner.go:221, `autoExpandWorker` runner.go:375, `validationWorker` runner.go:396). An unrecovered panic in any of these takes down the **entire worker process** (Go's per-goroutine panic semantics; `g11_worker_design.md` §1.3).

`dag_orchestrator` maps cleanly onto the **scheduling** half of the fix `g11_worker_design.md` §2.2/§2.3 already designed, without contradicting any of its decisions:

- **Per-job panic firewall (defect #2, job-execution granularity).** `g11_worker_design.md` §2.2 proposes wrapping `executeJob`'s handler dispatch in its own `recover()` (T3/T9 in its test table) so a panicking job becomes a recorded failure rather than a process death. `dag_orchestrator`'s `runOnce` (dag.go:397-405) already does exactly this at the node-execution boundary, tested (implicitly, via the module's own race-clean suite) rather than newly hand-written. If the worker's queued job-dispatch path (`executeJob`, runner.go:280-314) is expressed as a one-node-per-job `dag.Node` (trivial: `&dag.FuncNode{NodeID: job.ID.String(), Fn: func(ctx, in) (dag.Output, error) { return r.dispatchHandler(ctx, job) }}`) and run through `dag.NewScheduler().Run(ctx, d, dag.Options{Parallelism: N, Retry: dag.RetryPolicy{...}})`, the per-job panic-safety and retry-with-backoff (`RetryPolicy`, dag.go:54-65, 407-428) `g11_worker_design.md` §2.2 designs from scratch are obtained from an already-tested library instead, and `executeJobWithRetry`'s current hand-rolled fixed exponential backoff (runner.go:236-277, no parallelism cap at all — jobs are drained one at a time from `jobChan` by a single `processJobQueue` goroutine, runner.go:220-233) gains a real concurrency cap it does not have today.
- **Recursive auto-expand fan-out (defect #1, the auto-expand cycle specifically).** `g11_worker_design.md` §2.3 designs `runAutoExpandCycle` as a flat sequential loop over `GetMissingSkills` entries, calling `autoExpand.Run` once per gap up to `MaxNewSkillsPerRun`, with no parallelism and no structured record of which expansion produced which new skill. `dag_orchestrator`'s `Expandable` (dag.go:39-41) is a direct structural match for this cycle's actual shape: build one `dag.Node` per currently-known gap (`Execute` = call `r.autoExpand.Run(ctx, entry.SkillName, maxDepth)`), and have that node's `Expand(ctx, out)` return new `dag.Node`s for any dependency the expansion discovered still missing — the scheduler's own re-`dispatch()` (dag.go:306-320) then drives the recursive discovery **up to `MaxDepth`** (enforced by the node closure decrementing a depth counter and returning `nil, nil` from `Expand` at depth 0, i.e. `g11_worker_design.md`'s existing `MaxDepth`/`MaxNewSkillsPerRun` bounds are preserved as node-construction parameters, not weakened) — under a real `Options.Parallelism` cap and with `Result.Lineage` (dag.go:80-86) recording exactly which expansion triggered which follow-on skill for free, where today's code has no such record (`processed++` only, runner.go:479).
- **What is explicitly NOT replaced (honest boundary):**
  - **The ticker-driven perpetual polling loops themselves** (`autoExpandWorker`'s `time.NewTicker(5 * time.Minute)`, runner.go:378; `validationWorker`'s 2-minute ticker, runner.go:399; `registryReviewWorker`'s configurable-hour ticker, runner.go:417-423) are **outside** `dag_orchestrator`'s scope. `Scheduler.Run` is a single bounded execution over one `*DAG` that returns once every reachable node is done/failed/skipped (dag.go:185, 323-343) — it is not itself a perpetual service. The worker would call `dag.NewScheduler().Run(...)` fresh, once, on **every tick**, building a new `*DAG` each time from the gap-set/draft-set current at that tick. `g11_worker_design.md` §2.2's `supervise()` panic-firewall-plus-restart-with-backoff wrapper (runner.go-level, wrapping the *goroutine that ticks and re-invokes the scheduler*, not the nodes inside one scheduler run) is therefore **still required** — `dag_orchestrator` protects a node's `Execute`, not the outer polling goroutine that decides when to build and run the next DAG. Composing them: `supervise(ctx, "autoexpand_cycle", r.autoExpandLoop)` wraps the ticker loop; inside each tick, `r.runAutoExpandCycle` builds a fresh `*DAG` from the current gaps and calls `scheduler.Run`; a panic *inside* a node is caught by `dag_orchestrator`'s own `runOnce` and becomes a per-node `Result.Failed` entry, never reaching `supervise`'s outer recover at all (defence in depth, not redundant — the two recover layers guard different things: one node's business logic vs. the goroutine plumbing around building/running the DAG itself, e.g. a nil `r.autoExpand` pointer dereferenced while *constructing* the node list before `Build`/`Run` is even called).
  - **The Postgres single-owner advisory lock** (`g11_worker_design.md` §3, `pg_try_advisory_lock`) has no `dag_orchestrator` analogue — it remains exactly as designed, guarding which worker *replica* is allowed to run a given cycle's `dag.NewScheduler().Run(...)` at all, upstream of the scheduler.
  - **All skill-graph business logic** — `autoexpand.Pipeline.Run`, `validation.Pipeline.Validate`, `GetCoverageStats`/`GetMissingSkills`/`ListSkills`, persisting new draft skills, updating `skill_registry` — stays exactly as `g11_worker_design.md` §2.3 designed it, now simply invoked from inside `dag.Node.Execute`/`Expand` closures instead of a flat `for` loop. `dag_orchestrator` has zero awareness of skills, Postgres, or LLMs, by design (`dag.go:1-9` package doc: "agent-free ... operates on opaque node payloads so it is reusable by any project").
  - **The typed-`CoverageStats` fix for defect #2's OTHER instance** (`runRegistryReview`'s bare `.(int)`/`.(string)`, runner.go:518-519) is unrelated to scheduling and stays exactly as `g11_worker_design.md` §2.1 designed (a `GetCoverageStats` typed accessor on `*skill.Store`) — `dag_orchestrator` has no bearing on this half of defect #2 at all; it is a store-layer API-safety fix, not a scheduling concern.
  - **The queued on-demand job path's simplicity** — migrating `executeJob`'s four `handleAutoExpand`/`handleValidate`/`handleCodeAnalysis`/`handleRegistryReview` dispatch (runner.go:290-301) onto per-job single-node DAGs is an **optional** consistency choice (§2.1 sketch above), not mandatory: the ticker-driven cycles are where the real fan-out/parallelism win is (multiple independent gaps/drafts per tick); the on-demand queue today already serialises one job at a time through a single `processJobQueue` goroutine (runner.go:220-233) with no correctness defect from doing so, only a missed opportunity for concurrency the design may take up later without urgency.

### 2.3 Verdict: INCORPORATE-as-submodule (scoped)

**INCORPORATE-as-submodule**, scoped exactly to the scheduling machinery identified in §2.2: replace `g11_worker_design.md`'s hand-designed `supervise()`-wraps-nodes-directly + flat sequential auto-expand loop with `dag_orchestrator`'s `Scheduler.Run`/`FuncNode`/`Expandable`/`RetryPolicy`/`Options.Parallelism` driving each tick's gap-set and draft-set, while **keeping** (not replacing) the per-goroutine `supervise()` firewall around the outer ticker loops, the Postgres single-owner advisory lock, and all business-logic wiring exactly as `g11_worker_design.md` §2/§3 already designed them. This is a strict superset of correctness (real parallelism cap + retry + panic-per-node + lineage, all already tested) obtained from a genuinely complete, non-stub, leaf dependency, at the cost of one new dependency import and re-expressing two cycle bodies (`runAutoExpandCycle`, and optionally `executeJob`) as `dag.Node` closures instead of flat loops.

This **confirms and narrows** `research/helix_family_reusable_practices.md` §8A's flag: that pass named `dag_orchestrator` as "a direct candidate for G11 worker scheduling ... a follow-on pass should design its wiring against `research/g11_worker_design.md`" without examining either file's source; this pass has now done so and finds the fit is real (§2.2), not merely plausible, and identifies the precise boundary of what it does and does not replace.

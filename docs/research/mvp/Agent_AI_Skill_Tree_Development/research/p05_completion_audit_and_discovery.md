### A.2 — NEW finding: `Store.Search` (the wired, live search path) is not vector/hybrid search despite its own doc-comment; the correct vector implementation is dead code with zero callers in all history

- **Proposed ID:** G29
- **Category:** gap / naming-bluff (same class as G02's "sandbox" misnomer, G12's "tree-sitter" stub)
- **Severity:** HIGH
- **Evidence:**
  - `internal/skill/store.go:50-51`: `// Search performs a hybrid search combining vector similarity and text matching.` / `func (s *Store) Search(ctx context.Context, query string, limit int) ([]models.SearchResult, error) {` — the doc-comment makes an explicit vector-search claim.
  - `store.go:52-115`: the implementation is `similarity(...)` (pg_trgm) + `ILIKE` only, in two successive SQL statements (primary trigram match, then an `ILIKE`-only fallback on zero rows). **No embedding, no `<=>` operator, no call to any embedder anywhere in this function.**
  - `store.go:574-609`: a **separate**, correctly-implemented method, `Store.VectorSearch(ctx, embedding []float32, limit int)`, exists in the same file and correctly performs `ORDER BY s.embedding <=> $1` cosine-distance search via pgvector.
  - `grep -rn '\.VectorSearch\('` across all 53 non-test files: **zero matches.** `git log --oneline -S".VectorSearch(" --all -- project`: **zero commits** — this method has never had a caller in the repository's recorded history.
  - `Store.Search` (the trigram-only path) **is** wired, into three call sites: `internal/mcp/tools.go:56` (the MCP `skill_search` tool — the tool the system's own prompt text tells agents to "Use this FIRST"), `cmd/server/main.go:207` (the live ad-hoc REST server's `/api/v1/skills/search` route), and `internal/validation/pipeline.go:531,549` (the "zero-bluff" jury pipeline's own duplicate-skill / existing-dependency checks — once G03's wiring lands, the pipeline's duplicate-detection will *also* rely on trigram/ILIKE text matching, not semantic similarity, silently weakening its ability to catch a semantic duplicate phrased differently).
  - Contrast: `internal/api/search_handler.go` (the *hardened*, currently-dead-per-G01 REST path) correctly calls `s.pool.SearchSkills(...)`, which delegates to the real vector-capable `internal/db/vector.go` implementation — so a **third**, correct, vector-capable code path exists, but it too is unreachable while G01's dead-server consolidation (its own tracked "O3" residual) remains open.
- **Why it matters:** the system's core value proposition (per `SPEC.md`, `REQUIREMENTS.md`, and the 768-dimension embedding infrastructure G10 already scrutinises) is semantic/vector search. The one search path that actually runs today — for the primary AI-agent-facing MCP tool, for the live REST route, and for the validation pipeline's own integrity checks — never touches an embedding, contradicts its own doc-comment, and sits beside a correct, tested-looking sibling method that has never been called. This is the same "two/three rival implementations, the wrong one is wired" structural pattern the register already names for the API server (G01) and for validation/autoexpand (G03), recurring a third time for search specifically, but none of G01/G03/G09 evidence this particular defect.
- **Proposed fix direction:** wire `Store.Search` to call `Store.VectorSearch` (with an embedder resolved from config) as the primary path, falling back to trigram/ILIKE only when no embedder is configured — and say so honestly in the result set (never silently), or retarget MCP `skill_search` + `cmd/server/main.go`'s route + the validation pipeline's duplicate-check directly at the already-correct `Pool.SearchSkills`/`Store.VectorSearch` once G01's consolidation lands. Correct the doc-comment to state actual behaviour until the fix lands.

### A.4 — NEW finding: unchecked type assertion inside the panic-recovery handler itself, safe only by exact middleware-ordering coincidence

- **Proposed ID:** G34 *(kept in numeric sequence with the search/analysis findings above; see final numbering table in A.7)*
- **Category:** weakness / fragility
- **Severity:** LOW–MEDIUM
- **Evidence:**
  - `internal/api/middleware.go:181,184`: `rid, _ := c.Get("request_id")` then `zap.String("request_id", rid.(string))` — a **bare** type assertion (no `,ok`) inside `Logger()`.
  - `middleware.go:256,258-268`: the same bare `rid.(string)` pattern appears **twice** inside `Recovery()` — the middleware whose entire job is to catch and gracefully report panics.
  - `c.Get("request_id")` returns `(nil, false)` whenever the key was never set on the request context; `nil.(string)` panics with "interface conversion: interface {} is nil, not string".
  - Verified this is **currently not reachable** in practice: `internal/api/server.go:128-153` (`setupMiddleware`) registers global middleware in the exact order `Recovery()` → `RequestID()` → `Logger()` → ... on every route registered afterward (there is no second router/route-group in the codebase that bypasses this global chain — confirmed by reading the full `SetupRoutes` route table, `server.go:157-217`), so `RequestID()` always populates `request_id` before any downstream handler (including `Logger`) can run. §11.4.194 requires this "effectively unreachable" claim be *proven*, not assumed — it is proven here by reading the actual registration order and the full route table, not inferred from the comment "Recovery must be first to catch panics from other middleware" (`server.go:129`).
  - However there is **no defensive `,ok` fallback anywhere**, so correctness depends entirely on that exact three-line ordering being preserved forever, across every future change, in every place a router is assembled (e.g., a future admin-only sub-router, or a route group built without the shared `setupMiddleware()` helper, would not carry this guarantee).
- **Why it matters:** a panic inside the panic-recovery callback itself (were the ordering ever broken) converts a graceful 500 into a raw, unrecovered second panic — the exact "fixed-but-not-working" verification gap class §11.4.108/§11.4.194 exist to catch, here found *before* it manifests rather than after.
- **Proposed fix direction:** use the `,ok` form with a `"unknown"` fallback in both `Logger()` and `Recovery()`; add a regression test that constructs a bare `gin.Engine` with only `Recovery()`+`Logger()` (no `RequestID()`) and asserts no panic.

- **Proposed ID:** G35
- **Category:** existing-bug / inconsistency (client↔server integration break, distinct from G01's server-side auth posture)
- **Severity:** HIGH
- **Evidence:**

- **Why it matters:** the two actual end-user-facing applications this system ships (the CLI, and the TUI — the reference clients ahead of the Aurora/HarmonyOS clients G15 discusses) send their API key in a header the server-side authentication middleware never inspects. - **Proposed fix direction:** either change `APIKeyAuth` to accept `Authorization: Bearer <key>` as an equally valid credential channel (documented, consistent with common REST convention) or change both `cmd/cli/commands/common.go` and `cmd/tui/api_client.go` to send `X-API-Key`. Whichever direction is chosen, add an integration test that exercises the actual CLI/TUI request-building code against a real `APIKeyAuth`-protected test server — the exact "test doesn't exercise the real header" gap that let this drift unnoticed.

- **Category:** weakness (same family as the already-filed G25's swallowed-`_ =`-Scan-error pattern, but a distinct file/function G25/G07 never evidenced)
- **Severity:** LOW (currently unreachable, confirmed dead code; would be HIGH-impact-on-data-fidelity if ever wired without fixing this first)
- **Evidence:**
  - `internal/skill/import_export.go:262-268` (inside `Store.ExportToTOML`): when a dependency's `DependsOnName` was not already populated, the code does `var name string` then `_ = s.pool.QueryRow(ctx, \`SELECT name FROM skills WHERE id = $1\`, dep.DependsOn).Scan(&name)` — the `Scan` error is discarded — followed by `depName = name`. On any DB error (connection blip, the dependency's target skill having since been deleted, etc.) `name` silently stays `""`, and an **empty string is appended into the exported TOML's `requires`/`extends`/`recommends` array** (`import_export.go:270-277`) rather than the error surfacing or the entry being omitted.
  - `grep -rn "ExportToTOML"` across every non-test `.go` file: the only occurrence anywhere is the definition itself. `git log --oneline -S"ExportToTOML" --all -- project`: exactly one commit (`a2f2631`, the single squashed introduction of the whole project tree) — this function has never had a caller in this repository's recorded history, confirming it is dead code by the same git-evidenced standard applied to `Store.VectorSearch` in A.2.
  - The REST server's actual export path uses a **different**, correctly-implemented function, `exportToTOMLWrapper` in `internal/api/skills_handler.go:590` (a package-level function in `internal/api`, not this `Store` method) — the one G07 already cites as correctly writing `DependsOnName`.
- **Why it matters:** this is not live today, so it is not itself a shipping defect — but it is exactly the kind of pre-built, plausible-looking, never-audited method a future change could reach for (it already exists, is exported, and is named exactly what a "wire up MCP export" task would look for) and ship with a silent data-corruption defect intact. Flagging it now, before it is wired, is cheaper than finding it after.
- **Proposed fix direction:** either delete it with a §11.4.124-compliant citation of this exact git-history evidence (zero callers across all recorded history) in its own commit, after an explicit operator keep-or-remove decision per §11.4.122 — or, if it is intended as the future MCP export path, fix the swallowed error (return it, or omit the entry with a logged warning) before wiring it in.
